# Гайд По Админке

Этот документ описывает текущее состояние админки.  
Если что-то расходится с этим файлом, источником истины считается код и [README.md](/home/egor/Work/src/github.com/Jopoleon/invest-control-bot/README.md).

## Вход

Актуальная модель авторизации:

- вход через `/admin/login`;
- оператор вводит `ADMIN_AUTH_TOKEN`;
- после успешного входа создается server-side session;
- в браузер пишется `HttpOnly` cookie;
- на каждый запрос к `/admin/*` сессия валидируется сервером.

Дополнительно:

- для machine-to-machine запросов можно использовать `Authorization: Bearer <ADMIN_AUTH_TOKEN>`;
- если `ADMIN_AUTH_TOKEN` пустой и среда `local`, админка открыта без логина.

## Основные экраны

### `/admin/connectors`

Экран управления тарифами.

Здесь можно:

- создать коннектор;
- включить или выключить тариф;
- удалить тариф, если он не используется в истории;
- скачать CSV;
- открыть ссылку в Telegram;
- открыть ссылку канала.

### `/admin/users`

Список пользователей.

Здесь можно:

- найти пользователя по `user_id`, linked messenger identity или поиском;
- открыть карточку пользователя;
- увидеть общий статус автоплатежей.

### `/admin/users/view`

Карточка пользователя.

Здесь видно:

- профиль;
- подписки;
- платежи;
- аудит;
- согласия;
- recurring summary;
- историю recurring-consents.

Доступные действия:

- отправить сообщение пользователю;
- отправить ссылку на оплату;
- вручную отключить подписку;
- запустить `rebill`;
- открыть публичную страницу отключения автоплатежа.

### `/admin/billing`

Экран платежей и подписок.

Здесь можно:

- смотреть платежи;
- смотреть подписки;
- фильтровать данные;
- выгружать CSV;
- видеть summary по текущей выборке.

### `/admin/events`

Аудит действий пользователей и операторов.

Это server-side пагинация по backend-данным, а не client-side таблица.

### `/admin/churn`

Операционный экран проблемных подписок и оплат.

Здесь видно:

- failed payments;
- pending payments;
- expired или revoked subscriptions;
- recurring/autopay state;
- retry state;
- время последней попытки rebill.

Доступные действия:

- открыть карточку пользователя;
- отправить ссылку на оплату;
- вручную запустить rebill;
- экспортировать CSV.

### `/admin/legal-documents`

Реестр юридических документов.

Поддерживаются:

- `offer`;
- `privacy`;
- `user_agreement`.

Можно:

- создать новую версию;
- редактировать существующую;
- включать и выключать документ;
- удалять документ;
- получать публичные ссылки.

### `/admin/sessions`

Экран активных admin-сессий.

Можно:

- посмотреть все активные браузерные сессии;
- понять, какая сессия текущая;
- вручную завершить подозрительную или зависшую сессию.

### `/admin/help`

Справочный экран по интерфейсу.

## Коннекторы

Коннектор — это тариф или продукт, который покупает пользователь.

### Основные поля

- `Name`: название тарифа в боте и админке.
- `Description`: описание тарифа.
- destination/access settings: Telegram/MAX destination для выдачи доступа.
- `Channel URL`: прямая ссылка на канал или страницу доступа; приоритетнее для UI, если задана.
- `Price RUB`: стоимость подписки в рублях.
- `Period mode`: модель периода (`duration`, `calendar_months`, `fixed_deadline`).
- `Period seconds`: длительность для `duration`, например `2592000` для 30 дней или `180` для короткого smoke-теста.
- `Period months`: количество календарных месяцев для `calendar_months`.
- `Fixed ends at`: дата окончания для `fixed_deadline`.
- `Start payload`: deeplink payload для `/start ...`; если пусто, генерируется автоматически.
- `Offer URL`: собственная ссылка на оферту для конкретного тарифа.
- `Privacy URL`: собственная ссылка на политику ПДн для конкретного тарифа.

### Что обязательно

- обязательно заполнить `Name`;
- обязательно заполнить `Price RUB`;
- нужно задать destination или channel URL, чтобы оператор и пользователь понимали, куда ведет доступ.

## Платежные режимы

В проекте сейчас есть два режима:

### `mock`

Используется для локального тестирования без реального провайдера.

### `robokassa`

Рабочая интеграция с Robokassa.

Поддерживаются:

- первый платеж;
- callback на успешную оплату;
- callback на неуспешную оплату;
- recurring opt-in flow;
- ручной и автоматический `rebill`;
- публичные страницы оформления и отключения автоплатежа.

## Юридические документы

Если у коннектора не указаны свои `Offer URL` и `Privacy URL`, бот и public pages могут брать активные документы из реестра.

Поэтому для корректного платежного и recurring flow желательно иметь опубликованные:

- оферту;
- политику конфиденциальности;
- пользовательское соглашение.

## Аудит

Экран `/admin/events` показывает ключевые события:

- `/start`;
- принятие условий;
- создание платежа;
- отправку платежной ссылки;
- успешную оплату;
- failed payment;
- включение и отключение автоплатежа;
- recurring notices;
- действия админки.

## Что проверять оператору в первую очередь

### Если пользователь не получил доступ

Проверь:

- есть ли `paid` платеж;
- создалась ли подписка;
- активна ли она;
- какой канал указан в коннекторе.

### Если пользователь говорит, что автоплатеж отключен или включен неправильно

Проверь:

- recurring summary в карточке;
- историю `Recurring consents`;
- `auto_pay_enabled` у конкретных подписок;
- аудит по `autopay_enabled` / `autopay_disabled`.

### Если не прошел rebill

Проверь:

- экран `/admin/churn`;
- last rebill status;
- retry state;
- failed payment notice;
- можно ли отправить ссылку на ручную оплату.

## Что не стоит считать источником истины

Не стоит ориентироваться только на:

- success-страницу в браузере;
- сообщение пользователя;
- внешний вид кнопок в боте.

Источником истины считаются:

- записи в платежах;
- записи в подписках;
- callback от платежного провайдера;
- аудит событий.
