# Регуляторика хранения данных (РФ): выжимка для сервиса

Дата: 2026-03-10
Статус: рабочая памятка для проектирования
Важно: это не юридическое заключение. Перед запуском в прод требуется проверка с юристом/ДПО.

## 1) Базовые требования по ПДн
- Основной закон: 152-ФЗ «О персональных данных».
- Для нашего сервиса ПДн: ФИО, телефон, email, Telegram ID/username, история действий/платежей (если позволяет идентифицировать лицо).
- Нужны: правовое основание обработки, информирование субъекта, политика ПДн, фиксация согласий, цели и сроки хранения.

Что это означает для нас:
- В боте фиксировать факт акцепта оферты/согласия на ПДн (дата/время, user id, connector id).
- Хранить только необходимые поля и только на нужный срок.
- Реализовать процессы удаления/обезличивания по истечении срока хранения.

## 2) Локализация ПДн граждан РФ
- 242-ФЗ ввел требование первичной записи/накопления ПДн граждан РФ в базах данных на территории РФ.
- При использовании внешних сервисов (логирование, аналитика, SaaS) нужно отдельно проверять, не нарушается ли правило локализации.

Что это означает для нас:
- Основная БД с ПДн должна быть размещена в РФ.
- Бэкапы и реплики с ПДн тоже учитывать как хранение.

## 3) Уведомление Роскомнадзора
- Оператор ПДн обычно подает уведомление в РКН до начала обработки (есть исключения, но их надо подтверждать отдельно).
- Есть утвержденная форма уведомления (Приказ РКН №180).

Что это означает для нас:
- До прод-запуска проверить обязанность подачи и, при необходимости, подать уведомление.
- Внутри проекта иметь карточку оператора: цели обработки, категории ПДн, сроки, меры защиты, трансграничная передача.

## 4) Меры защиты информации
- Требования к защите ПДн задаются, в том числе:
- Постановлением Правительства РФ №1119.
- Приказом ФСТЭК России №21.
- Приказом РКН №140 от 19.06.2025 (требования и методы по обезличиванию ПДн).

Что это означает для нас (минимум):
- Разграничение доступа (админка только для уполномоченных).
- Учет действий админов и критичных операций (audit log).
- Защита каналов передачи (TLS).
- Защита хранения (шифрование чувствительных полей/бэкапов, mask в логах).
- Регламент реагирования на инциденты и утечки.
- Для сценариев аналитики и разработки определить процесс обезличивания выгрузок/логов.

## 5) Ответственность
- За нарушения правил обработки ПДн применяются санкции по ст. 13.11 КоАП РФ.

Что это означает для нас:
- Документация и процессы не менее важны, чем код.
- Нужны формальные документы: политика ПДн, положение об обработке ПДн, модель угроз/меры защиты (по необходимости), регламент удаления данных.

## 6) Практический чеклист для проекта
1. Утвердить цели обработки и состав ПДн по каждой сущности.
2. Зафиксировать сроки хранения и правила удаления/обезличивания.
3. Включить хранение факта согласия (кто/когда/на что согласился).
4. Хранить БД и бэкапы в РФ.
5. Подготовить/опубликовать политику ПДн и тексты согласий.
6. Проверить обязанность уведомления РКН и подать при необходимости.
7. Ограничить и аудитировать доступ к админке и данным.
8. Убрать ПДн из технических логов (или хранить ограниченно и защищенно).
9. Прописать процедуру инцидентов и запросов субъектов ПДн.
