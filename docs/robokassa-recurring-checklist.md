# Robokassa Recurring Checklist

Дата: 2026-03-20  
Источник: официальная документация Robokassa + переписка с поддержкой

## Вывод

Recurring у Robokassa не считается доступным "по умолчанию".  
Для этого проекта услуга уже была отдельно согласована и включена магазином Robokassa после подготовки product/compliance части.

Это означает:

- recurring можно тестировать end-to-end;
- но production readiness все равно зависит от качества юридических текстов, UX и операционного контроля.

## Что уже подтверждено

- Для recurring используем self-integration сценарий:
  - первый платеж с `Recurring=true`
  - повторные списания через `Merchant/Recurring`
  - успешность списания подтверждается только через `ResultURL` / `ResultUrl2`
- Ответ `OK+InvoiceID` от `Merchant/Recurring` сам по себе не означает успешную оплату.
- Поддержка Robokassa попросила:
  - подготовить юридические документы
  - подготовить UX формы оплаты
  - после этого отдельно сообщить `Shop ID` для включения recurring

## Требования Robokassa к продукту

### 1. Юридические документы

Должны быть доступны и опубликованы:

- публичная оферта
- политика конфиденциальности
- пользовательское соглашение
- согласие на обработку персональных данных

### 1.2. Что обязательно должно быть в оферте для recurring

- условия подписки: период, стоимость
- порядок списания средств: дата, время
- способы отмены подписки
- порядок возврата средств
- условия изменения стоимости
- отдельный раздел про автоматические списания

### 2. Форма оплаты / UX

Обязательно:

- отдельный чекбокс согласия на автоматические списания
- чекбокс не должен быть проставлен по умолчанию
- рядом должен быть текст:
  - `Я согласен на автоматические списания согласно условиям оферты`
- ссылка на оферту должна быть кликабельной
- пользователь должен видеть периодичность списаний

Важно:

- нужно хранить историю согласий пользователя на автосписания
- нельзя делать скрытые платежи
- нельзя делать сложную отмену подписки
- нельзя включать автопродление без уведомления
- нельзя менять цену без предупреждения

## Что уже есть в проекте

- реестр оферт/ПДн в админке
- отдельный тип документа `user_agreement` в legal registry
- публичные URL документов
- хранение обычных согласий пользователя на оферту/ПДн
- versioning согласий в `user_consents`
- отдельная таблица `recurring_consents` под историю opt-in на автосписания
- базовая архитектура recurring/self-integration в backend
- публичная recurring checkout page `/subscribe/{start_payload}` с recurring opt-in текстом и legal links
- публичная cancel page `/unsubscribe/{token}` без авторизации с простым отключением автоплатежа
- cancel-link встроен в bot flow как публичная точка отключения автосписаний
- recurring-архитектура и UX уже собраны, магазин активирован со стороны Robokassa

## Чего еще не хватает до полноценной боевой готовности

### Бизнес / юридический слой

- дописать оферту под recurring-условия Robokassa
- явно описать порядок отмены подписки
- явно описать политику возвратов
- явно описать изменение стоимости

### Продукт / UX

- дополнительно согласовать тексты public recurring checkout/cancel pages под финальный юридический контент
- убедиться, что support/Robokassa устраивает текущая форма страницы оформления подписки и страницы отмены
- добавить уведомления:
  - о скором списании / окончании подписки
  - о неудачном списании
  - об отключении доступа

### Backend / data

- хранение истории попыток rebill
- retry policy 3 попытки в течение 3-5 дней
- ручной fallback на обычную оплату по тому же коннектору

## Предлагаемый порядок внедрения

1. Подготовить юридический контент под recurring.
2. Добавить в продукт отдельный recurring opt-in и хранение истории согласий.
3. Доделать UX отмены автоплатежа.
4. Подтвердить readiness и отправить в Robokassa `Shop ID`.
5. Получить активацию recurring у магазина.
6. Включить `ROBOKASSA_RECURRING_ENABLED=true`.
7. Прогнать E2E на реальном тестовом flow.

## Для текущего проекта

На текущем этапе:

- recurring уже можно тестировать;
- первый платеж с opt-in должен уходить с `Recurring=true`;
- повторное списание подтверждается только через `ResultURL`;
- отключение автоплатежа должно работать по конкретной подписке, а не глобально по пользователю;
- operator UX должен позволять видеть consent history, retry state и проблемные rebill.
